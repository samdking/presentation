Branching

Three important parts of Git:

  1. The HEAD: your last commit
  2. The Index: stage for your next commit
  3. Working Directory: your files

git submodule update

Update the commit of my submodule inline with what the master repository says it should be on.

The master project does not care about anything more in the sub repository, other than the commit it is currently pointing at.

git reset HEAD --hard

HEAD points to the last snapshop, the current commit you are on. It will discard any changes staged in the index and revert the index back to the current commit (HEAD).

By default, this won't affect your working directory (the actual files). Use the --hard flag to physically reset the changes made to files following your last commit.

git reset HEAD~1

If you do a bad commit, you can undo it by resetting to HEAD~1. That is, The HEAD will point back at the commit before your last. Your changes will still be staged in the index, ready to commit again, or modify.

git reset --soft HEAD~1

git reset --mixed

git reset --hard HEAD~1

git checkout OBJECT <files>

Modify the files in your working directory to match that of OBJECT. This object will usually be identified by a branch name.

Checking out a commit hash means you are no longer on a branch, putting you in "Detached Head State". This is what Git is doing behind the scenes when updating a submodule.

Using -- as the object will use the current HEAD. It will discard any changes in your working directory.
